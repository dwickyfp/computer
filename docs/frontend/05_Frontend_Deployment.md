```bash

# 2. Install dependencies

# 3. Configure environment

# Edit .env.local ŌĆö set VITE_API_URL

pnpm dev
```

## 3. Environment Variables

| Variable       | Required | Default (dev)                  | Description                                                                   |
| -------------- | -------- | ------------------------------ | ----------------------------------------------------------------------------- |
| `VITE_API_URL` | Ō×¢       | `http://localhost:8000/api/v1` | Backend API base URL. Omit in production (uses `window.location.origin/api`). |

In production, `VITE_API_URL` should be **left unset**. The `getBaseUrl()` function in `repo/client.ts` automatically uses `window.location.origin + '/api'` when the variable is absent, allowing the frontend to work behind any domain with Nginx reverse proxying `/api` to the Backend.

If you need to point to a non-colocated Backend (separate domain), set:

```
VITE_API_URL=https://api.your-domain.com/api/v1
```

## 5. Vite Build Optimisations

`vite.config.ts` defines manual chunk splits for optimal caching:

| Chunk Name        | Contents                     |
| ----------------- | ---------------------------- |
| `tanstack-query`  | `@tanstack/react-query`      |
| `tanstack-router` | `@tanstack/react-router`     |
| `tanstack-table`  | `@tanstack/react-table`      |
| `charts`          | `recharts`                   |
| `flow-diagram`    | `@xyflow/react`              |
| `radix-ui`        | All `@radix-ui/*` primitives |

These stable vendor chunks are cached by browsers across deployments, significantly reducing repeat visit load times.

## 6. Nginx Configuration (Production)

The `docker/nginx.conf` configures Nginx as the reverse proxy for the full stack:

```nginx
server {
    listen 80;

    # Serve React SPA ŌĆö all unknown paths return index.html (SPA routing)
    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;
    }

    # Proxy API requests to Backend
    location /api/ {
        proxy_pass http://localhost:8000/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # Proxy WebSocket (pipeline status)
    location /ws/ {
        proxy_pass http://localhost:8000/ws/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

The `try_files $uri /index.html` directive is **critical** ŌĆö it ensures deep links (`/pipelines/42`) are always served `index.html` so TanStack Router can handle client-side navigation.

## 7. Docker Deployment

In the full-stack Dockerfile, the frontend is built at image build time:

```dockerfile
# Build stage
FROM node:20-alpine AS frontend-builder
WORKDIR /web
COPY web/package.json web/pnpm-lock.yaml ./
RUN npm install -g pnpm && pnpm install --frozen-lockfile
COPY web/ ./
RUN pnpm build

# Production stage ŌĆö copy dist to Nginx
COPY --from=frontend-builder /web/dist /usr/share/nginx/html
```

## 8. Netlify Deployment (Optional)

The repo includes `netlify.toml` for direct Netlify deployment of the frontend:

```toml
[build]
  base    = "web"
  command = "pnpm build"
  publish = "dist"

[[redirects]]
  from   = "/*"
  to     = "/index.html"
  status = 200
```

The redirect rule replicates the Nginx `try_files` behaviour for SPA routing on Netlify.

## 9. TanStack Router Code Generation

Routes are defined as files in `src/routes/`. The route tree is auto-generated by the Vite plugin:

```bash
# Route tree is auto-regenerated on every `pnpm dev` or `pnpm build`
# The generated file is:
src/routeTree.gen.ts   ŌåÉ DO NOT EDIT MANUALLY
```

When adding a new route:

1. Create `src/routes/_authenticated/<route-name>.tsx`
2. Export `Route = createFileRoute(...)({ component: MyPage })`
3. Vite plugin auto-updates `routeTree.gen.ts` on next save

## 10. Production Checklist

- [ ] `VITE_API_URL` unset (uses `window.location.origin/api`) or explicitly set to correct API domain
- [ ] `pnpm build` completes without TypeScript errors
- [ ] `pnpm lint` reports no errors
- [ ] Nginx `try_files $uri /index.html` configured for SPA routing
- [ ] CORS on Backend configured to allow the production frontend origin
- [ ] WebSocket upgrade headers configured in Nginx if pipeline-status WS is used
- [ ] Manual chunk splits in `vite.config.ts` not removed (browser caching strategy)
- [ ] `Content-Security-Policy` header configured in Nginx if hardening is required
- [ ] Sourcemaps excluded from production build (or uploaded to error tracking service only)
